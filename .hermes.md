# kaspa-pof-api Hermes Project Rules

These rules are loaded by Hermes when working inside this repo.

## Project purpose

Build `kaspa-pof-api` as a general-purpose npm package for proof-of-fairness applications using Kaspa/TN10/mainnet evidence.

Roulette is an example consumer, not the package's source of truth.

## Architecture rules

- Package-first: reusable proof/fairness primitives belong in the npm package.
- A VPS/server may exist as an optional adapter or convenience layer, but must not be the proof authority.
- The package must not depend on the roulette app; the roulette app must depend on the package.
- Do not copy old `kaspa-toccata-api/src/server.cjs` wholesale. Extract or rewrite pure functions first.
- Avoid inheriting architecture from older repos without fresh evidence.

## Proof/fairness rules

- No mock proof/result paths.
- No offline/local substitute proof paths unless the user explicitly authorizes a temporary exception.
- No static proof/result JSON fixture traps.
- No transaction simulation paths presented as success.
- Avoid dry-run terminology or flows for Kaspa/Toccata transaction work.
- Prefer honest claim levels that say exactly what was proven.
- Default mainnet direction: no-spend future-entropy verification first; optional paid anchoring only with explicit fee estimates, fee caps, and user/operator acknowledgement.

## Package design rules

- Keep APIs app-agnostic where reasonable.
- Use generic terms such as input ledger, commitment, entropy evidence, outcome deriver, proof bundle, and claim level.
- Roulette-specific UI/game logic belongs under `examples/roulette-poc/`.
- Core package modules should be deterministic and locally verifiable where possible.
- Verification must fail closed on missing, inconsistent, or unknown evidence.

## Development rules

- Stage work in small increments with evaluation points.
- Do not make unrequested UI/product changes.
- Discuss major layout or architecture pivots before implementing them.
- Use focused verification before reporting a change complete.
- Prefer industrial-grade verification language; do not describe proper verification work as a shortcut.
- If a profile/tool warning asks for temporary verification, call it focused temporary verification.
