# Next Phase Documentation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Add planning and specification documents that make the next `kaspa-pof-api` phase executable from a fresh Hermes profile.

**Architecture:** The docs should reinforce package-first proof-of-fairness architecture: npm package owns reusable proof/verification primitives; optional servers provide convenience only; roulette is an example consumer.

**Tech Stack:** Markdown docs in `/root/kaspa-pof-api`, npm package scaffold, roulette browser example.

---

## Task 1: Add next-phase plan

**Objective:** Define implementation workstreams and acceptance criteria.

**Files:**
- Create: `docs/NEXT_PHASE_PLAN.md`

**Steps:**
1. Document goal and non-goals.
2. Define workstreams for proof schema, primitives, outcome hooks, evidence validation, roulette local verification, HTTP adapter boundary, and mainnet anchoring design.
3. Include expected verification commands and first implementation sequence.

**Verification:**
- `docs/NEXT_PHASE_PLAN.md` exists.
- It mentions local package verification and not relying on `/v1/proofs/verify` as proof authority.

## Task 2: Add package specification

**Objective:** Define draft API/data model for the general-purpose npm package.

**Files:**
- Create: `docs/PACKAGE_SPEC.md`

**Steps:**
1. Document terminology: commitment, ledger, entropy, reveal, proof bundle, claim level.
2. Define draft public API exports.
3. Define portable `ProofBundle` shape and related evidence types.
4. Document verification rules and fail-closed behavior.
5. Document HTTP adapter and mainnet policies.

**Verification:**
- `docs/PACKAGE_SPEC.md` exists.
- It defines `ProofBundle`, `ClaimLevel`, and `VerificationResult`.
